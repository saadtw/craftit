1. MANUFACTURER VERIFICATION — REDESIGNED (2026-03-28 update)
─────────────────────────────────────────────────────────────────────────────
Original flow (Manufacturer Side.docx section 1.3): Manufacturers gated behind a
"Pending Verification" screen with no dashboard access until admin approved.

Previous interim decision: Gate removed — both unverified and verified had identical access.

CURRENT flow (now enforced at API level):

Signup collects (all required unless noted):
  - CEO / Contact Person Name *
  - Personal Email * (login email)
  - Personal Mobile Number *
  - Business Name *
  - Business Email *
  - Business / Office Phone *
  - Business Type * (sole_proprietorship | partnership | private_limited |
                     public_limited | ngo | other)
  - Business Registration Number (optional)
  - Business Description (optional)
  - City / State / Country (optional)
  - Manufacturing Capabilities (checkbox multi-select)
  - Materials Available (checkbox multi-select)

After signup — Unverified restrictions (enforced in API routes):
  - Can list up to 5 products (archived excluded from count)
  - Cannot upload 3D models on products
  - Cannot browse RFQs (GET /api/rfqs returns 403)
  - Cannot place bids (POST /api/bids returns 403)
  - Cannot create Group Buys (POST /api/group-buys returns 403)
  - Cannot receive orders (POST /api/orders/product returns 400 if manufacturer is unverified)
  - Shown with "Unverified" badge wherever their profile appears

Verification application — submitted via /manufacturer/settings:
  POST /api/verification-documents
  Fields accepted:
    - ntnNumber (NTN / STRN number text)
    - strnNumber
    - secpRegistrationNumber (SECP / Form-C number text)
    - documents[] — each with { type, url, filename, fileSize }
      Allowed doc types: ntn_certificate, secp_form_c, chamber_certificate,
                         business_license, tax_registration, certification,
                         insurance, bank_verification, identity, other
  On resubmission: documents are REPLACED (not appended).
  Manufacturer stays unverified until admin approves.

Admin review — PUT /api/admin/manufacturers/[id]/verify
  Actions: approve | reject | request_info
  All actions write to AdminLog (description + details fields).
  - approve: verificationStatus = "verified", notifies manufacturer
  - reject: reason required, notifies manufacturer
  - request_info: verificationStatus stays "unverified" on User,
    VerificationDocument.verificationStatus = "resubmission_required",
    7-day deadline set, fires "verification_info_requested" notification
    (NOT the rejection notification type — correct type is used now)

GET /api/verification-documents — manufacturer fetches their own status and doc record.

14. ORDER CANCELLATION & REFUND POLICY 
─────────────────────────────────────────────────────────────────────────────
Scope: Customer-initiated cancellation AFTER the manufacturer has accepted the order.
Pre-acceptance cancellation (existing) is unaffected — already fully implemented.

Basis: Industry standard for B2B custom manufacturing platforms (e.g. Alibaba Trade
Assurance, Xometry, Fictiv). Custom manufacturing differs from commodity goods —
once production starts, raw materials may be consumed and work-in-progress cannot
be easily reversed. Policy must balance buyer protection with fair treatment of
manufacturers who act in good faith.

─── CANCELLATION RULES BY ORDER STATUS ───────────────────────────────────────

Status: accepted
  Window: ≤ 48 hours since acceptance
  Who can cancel: Customer unilaterally (no manufacturer consent needed)
  Refund: 100% — Stripe refund on captured amount
  Rationale: Standard "cooling-off" window. Manufacturer is unlikely to have
  started procuring materials or begun any work this early.

Status: accepted
  Window: > 48 hours since acceptance
  Who can cancel: Customer requests cancellation; manufacturer must confirm
  Refund: 100% if manufacturer confirms. If manufacturer does not respond within
  72 hours, admin may intervene.
  Rationale: Manufacturer may have begun prep work. Mutual agreement is fair.
  If unresponsive, admin escalation protects the buyer.

Status: in_production
  Cancellation: NOT allowed unilaterally. Customer must open a Dispute.
  Refund: Admin-decided (full / partial / none) based on evidence of work done.
  Rationale: Materials likely consumed, labour underway. Unilateral cancellation
  would be unfair to the manufacturer. Dispute process ensures both sides are heard.

Status: shipped
  Cancellation: NOT allowed.
  Recourse: Dispute or return, handled case by case.

Status: completed
  Cancellation: NOT allowed.
  Recourse: Dispute only.

─── REFUND AMOUNTS ────────────────────────────────────────────────────────────

  Full refund (100%):
    - Cancellation within 48 hours of acceptance
    - Manufacturer confirms cancellation (any time before in_production)
    - Dispute resolved entirely in customer's favour
    Implementation: Stripe refund on the captured PaymentIntent

  Partial refund:
    - Dispute resolved as "partial_resolution"
    - Admin specifies percentage or fixed amount
    Implementation: Stripe supports partial refunds on a captured charge
    (refund amount passed to /api/payments/refund endpoint)

  No refund:
    - Dispute resolved in manufacturer's favour
    - Order completed and within no valid dispute window

  Platform fee:
    - For FYP: no platform fee retained on refunds. Full refund = 100% back to buyer.
    - Post-launch consideration: retain 2-3% to cover Stripe processing cost on
      cancellations after 48 hours (industry standard practice).

─── STRIPE IMPLEMENTATION NOTES ───────────────────────────────────────────────

  Existing infrastructure reused:
    - POST /api/payments/refund already handles full and partial Stripe refunds
      (used by dispute resolution). Cancellation refunds will call the same endpoint.
    - Order status will gain a new PATCH action: "cancel_request" (customer)
      and "cancel_confirm" (manufacturer) — separate from the status flow.

  New API needed (to be built):

15. SESSION INVALIDATION + FORCED LOGOUT
─────────────────────────────────────────────────────────────────────────────
Problem solved:
  Previously, if a user was deleted/deactivated/suspended in DB, an already-issued
  JWT session could still appear logged in on client pages until token refresh.

Implemented protection layers:

1) Server-side token revalidation (lib/auth.js)
  - JWT callback now revalidates token identity against User document every ~60s.
  - If user is missing, deactivated, suspended, or sessionVersion mismatches:
      token.sessionInvalid = true
      token.sessionInvalidReason = user_not_found | deactivated | suspended | version_mismatch
      token identity fields are cleared.
  - Session callback maps invalid token to:
      session.error = "SESSION_INVALID"
      session.errorReason = <reason>

2) Session versioning for global revocation (User model + auth)
  - User.sessionVersion is included in JWT at login.
  - On revalidation, DB value is compared to token value.
  - If DB sessionVersion changed, all previously issued JWTs become invalid
    automatically on next revalidation.

3) Immediate forced logout on admin suspend (admin users route)
  - In PATCH /api/admin/users/[id] action="suspend":
      user.sessionVersion increments before save.
  - This revokes all active sessions for that user (version mismatch path).

4) Realtime logout event channel (SSE)
  - New file: lib/sessionEmitter.js
      In-memory per-user subscriber map + publishSessionEvent().
  - New route: GET /api/auth/events
      Authenticated Server-Sent Events stream with heartbeat.
  - On suspend, admin route publishes:
      { type: "SESSION_INVALIDATED", reason: "suspended" }

5) Client auto-signout guards (SessionProvider)
  - SessionInvalidGuard: signs out when session.error === "SESSION_INVALID".
  - SessionRealtimeGuard: listens to /api/auth/events and signs out immediately
    when SESSION_INVALIDATED event arrives.
  - refetchInterval remains 0 (no periodic client polling), refetch on window focus kept.

6) API safety check for profile endpoint
  - GET /api/auth/me now returns 401 when session.error is SESSION_INVALID.

Production note:
  - Current SSE publisher is in-memory (single-instance process scope).
  - For multi-instance deployment, replace event backend with Redis pub/sub
    or another shared broker to broadcast invalidation across all instances.
